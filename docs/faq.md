# 常见问题

## `omp.h: No such file or directory`

说明当前编译器不支持 OpenMP，或没有正确安装开发环境。

先检查：

```bash
g++ -fopenmp --version
```

如果使用服务器 module 环境，可以查看：

```bash
module avail
module load gcc
```

## 优化版本比 baseline 还慢

这是正常现象，尤其是在小规模数据上。

常见原因包括：

- 创建线程或任务有额外开销；
- 任务划分太细；
- 同步太频繁；
- 内存访问成为瓶颈；
- 线程数超过了当前机器适合的范围。

报告中需要分析原因，而不是只写“没有优化成功”。

## 归并排序 task 写完后结果错误

优先检查：

- 左右区间是否重叠；
- `merge` 是否在两个子区间都排序完成后执行；
- 临时数组是否被多个任务写到同一区间；
- 是否忘记 `taskwait`；
- 阈值条件是否导致某些区间没有排序。

## Jacobi 优化后 checksum 不一致

优先检查：

- 是否错误更新了边界值；
- 是否在同一步迭代中混用了 `cur` 和 `next`；
- 是否在所有线程完成计算前就交换数组；
- 是否引入了不必要的共享临时变量；
- 是否比较了同样的 `n` 和 `steps`。

浮点数结果可能存在极小差异，报告中用绝对误差说明即可。

## 大规模运行太久怎么办

先确保小规模和中规模完成。大规模如果无法完成，需要在报告中写清楚：

- 使用的机器配置；
- 实际运行命令；
- 已经运行了多久；
- 是内存不足、时间过长，还是程序报错；
- 当前能完成的最大规模是多少。

## 可以使用 MPI 吗

可以作为拓展，但基础要求以 OpenMP 优化为主。

如果使用 MPI，需要额外说明：

- 进程数；
- 数据如何分块；
- 进程之间如何交换边界或汇总结果；
- 通信开销对加速比有什么影响。
